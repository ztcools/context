## 二、新发现问题

### P2 — 代码质量

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 1 | **连锁反应** | `graph-handlers.ts:98-104` | **`deleteNodesByFile` 修复不完整**：只在 `specificFiles && specificFiles.length > 0` 时调用。当 `mode='incremental'`（git diff 自动检测变更文件）时，`specificFiles` 为 undefined，`deleteNodesByFile` 不执行。增量模式仍产生 stale 节点和边。修复应覆盖 `mode === 'incremental'` 路径。 |
| 2 | **新发现** | `handlers.ts:440-568` | **`forceReindex` 不取消旧 `indexingTasks`**：当 codebase 正在后台索引时触发 `forceReindex`，`handleIndexCodebase` 仅清除 snapshot 状态和 collection，但未调用旧 `indexingTasks` 的 `controller.abort()`。旧任务继续运行，会在清除后的 collection 中写入数据，导致数据不一致。 |

### P3 — 边界情况

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 3 | **新发现** | `sync.ts:189,218` | **`handleSyncIndex` 重复获取 codebaseInfo**：先用 `getCodebaseInfoByIdentity(identity)` 获取了一次，后又用 `getCodebaseInfo(codebasePath)` 重复获取。第二次调用做了不必要的 localPath→identity 解析，信息冗余。 |
| 4 | **新发现** | `graph-store.ts:245` | **`findNodes` 结果循环中 N+1 查询 degree**：`getNodeDegree(nodeId)` 对每个结果行执行 2 次 COUNT 查询（inDegree + outDegree）。当 limit=200 时产生 400 次额外 SQL 查询。可改为 JOIN 子查询或批量查询。 |
| 5 | **新发现** | `graph-handlers.ts:426` | **`handleDetectChanges` git diff 分支错误吞咽**：`git diff ${baseBranch}...HEAD` 失败时 catch 块静默回退到 `git diff HEAD`。用户无法知晓原始命令因分支名错误而失败，可能得到非预期的 diff 结果。 |
| 6 | **新发现** | `sync.ts:305` | **`setInterval` 回调中 `handleSyncIndex()` 无 await**：虽然已添加 `.catch()`，但 `handleSyncIndex()` 是 async 函数，`setInterval` 回调不是 async。如果上一个周期的 sync 尚未完成，下一个周期会立即启动，可能导致并发 sync。 |
| 7 | **新发现** | `graph-handlers.ts:849` | **`scanFiles` 忽略列表不完整**：缺少 `.venv`、`vendor`、`target`、`coverage`、`.cache`、`.idea`、`.vscode`、`.circleci`、`bin`、`obj`、`out`、`tmp`、`temp`、`.tox`、`.mypy_cache`、`.pytest_cache`、`.turbo` 等常见目录。 |

## 四、结论

前七轮 52 个问题全部修复（P3 除外）。第八轮发现 7 个新问题，其中 2 个为连锁反应，5 个为之前未排查到的新 bug。

**重点关注：**
- **P2-1 连锁反应**：`deleteNodesByFile` 修复只覆盖了 `specificFiles` 路径，`mode='incremental'`（git diff 自动检测）路径未覆盖。需将条件改为 `if (specificFiles || mode === 'incremental')`。
- **P2-2 新发现**：`forceReindex` 不取消旧 `indexingTasks`，旧任务在清除后的 collection 中写入脏数据。需在 `forceReindex` 时调用 `controller.abort()` 并 await 旧任务。
- **P3-6 新发现**：`setInterval` 并发 sync 问题，可能导致多个 sync 周期同时运行。建议在 `handleSyncIndex` 入口处加 `isSyncing` 检查（已有），但 `setInterval` 回调应 await 或加防重入保护。

累计 59 个问题，0 个 P0。代码质量持续提升，核心功能稳定性良好。
