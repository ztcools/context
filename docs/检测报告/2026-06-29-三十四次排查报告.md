## 二、图索引"进度为 0"根因分析 🔴

### 2.1 现场证据

| 项目 | 状态 |
|------|------|
| 图数据库主文件 `knowledge-graph.db` | 40KB（空库） |
| WAL 文件 `knowledge-graph.db-wal` | **3MB**（异常大） |
| SHM 文件 | 32KB |
| WAL checkpoint 后 | 0 节点，0 边 |
| 快照文件 | 显示 `codebase-memory-mcp` 为 `indexing`（45%） |

### 2.2 根因: WAL 累积未提交事务

```
旧的一次图索引在 beginTransaction() 后崩溃/中断
     ↓
SQLite WAL 积累了 3MB 的未提交数据
     ↓
MCP 重启时 SQLite 自动回滚未提交事务
     ↓
WAL 内容被丢弃，数据库保持空状态
     ↓
下次索引又从头开始 → 重复崩溃 → 循环
```

**关键代码路径** (`graph-handlers.ts:113-172`):

```typescript
this.store.beginTransaction();
// 全量索引: 逐文件 fs.readFileSync + extract + upsertNode + upsertEdge
// 如果项目有 5000+ 文件，这个过程可能持续 30-120 秒
// 期间所有数据写入 WAL，未提交
for (const filePath of files) {
    const source = fs.readFileSync(filePath, 'utf-8');
    const result = this.extractor.extract(source, {...});
    // ... upsert nodes and edges
}
// 交叉文件调用解析（可能再耗时 10-30 秒）
crossEdges = this.resolveCrossFileCalls(project);
// 只有到这里才 COMMIT
this.store.commitTransaction();
```

**问题**: 整个图索引在一个**单一事务**中完成。如果在此期间进程被 kill、OOM、或 MCP 超时断开，事务回滚，所有数据丢失。对于大项目，这个时间窗口可能长达数分钟。

### 2.3 次要因素: `setImmediate` 中的静默失败

```typescript
// handlers.ts:367
setImmediate(async () => {
    try {
        // ... 图索引
        await this.graphToolHandlers!.handleIndexRepository({...});
    } catch (e: any) {
        console.warn(`[INDEX] Graph indexing failed (non-fatal): ${e.message}`);
    }
});
```

`handleIndexRepository` 在出错时**不抛异常**，而是返回 `{ content: [{ type: 'text', text: 'Error...' }] }`。因此 `setImmediate` 的 catch 块不会触发，错误只存在于返回对象的 text 中但**无人读取**。图索引失败完全静默。

### 2.4 次要因素: 向量已索引时图索引被跳过

```typescript
// handlers.ts:353-357
const vectorResult = await this.handleIndexCodebase(args);
if (vectorResult.isError) {
    return vectorResult;  // ← 直接返回，图索引不会执行
}
```

当向量索引已存在（非 force 模式），`handleIndexCodebase` 返回 `isError: true`，`handleIndex` 直接返回，**完全跳过图索引**。这意味着如果用户先索引了向量，图索引永远不会被触发，状态永远显示 0。

## 三、新发现 Bug

### Bug 10: 图索引使用单一事务，大项目崩溃风险 🔴 高

**文件**: `packages/mcp/src/graph-handlers.ts:113-172`  
**严重程度**: 高 — 导致图索引数据丢失和"进度为 0"

**问题**: 全量图索引（`beginTransaction` → N 个文件处理 → `commitTransaction`）在一个事务中完成。对于大项目，事务持续时间可能很长。如果进程在此期间被中断，所有未提交数据丢失。

**修复建议**: 改为按文件粒度的事务（每处理一个文件或一批文件就 commit），使部分进度可恢复。

### Bug 11: `handleIndexRepository` 错误被静默丢弃 🔴 高

**文件**: `packages/mcp/src/handlers.ts:394-398`  
**严重程度**: 高 — 图索引失败无感知

**问题**: `handleIndexRepository` 出错时返回 `{ content: [{ type: 'text', text: 'Error...' }] }` 而非抛出异常。`setImmediate` 回调中 `await` 正常返回，catch 不触发，错误信息无人读取。

**修复建议**: 检查返回值中的错误信息并记录日志，或让 `handleIndexRepository` 在出错时抛出异常。

### Bug 12: 向量已索引时图索引被跳过 🟡 中

**文件**: `packages/mcp/src/handlers.ts:353-357`  
**严重程度**: 中 — 用户永远看不到图索引结果

**问题**: 当向量索引已存在（非 force），`handleIndexCodebase` 返回错误，`handleIndex` 直接返回，完全跳过图索引。用户可能先索引了向量（通过旧版本或直接调用 `index_codebase`），之后图索引永远不会运行。

**修复建议**: 将图索引启动移到 `isError` 检查之前，或让 `handleIndexCodebase` 在"已索引"时返回非错误状态。

## 七、结论

### 7.1 图索引"进度为 0"根因

**直接原因**: 图索引使用单一事务（`beginTransaction` → 全部文件处理 → `commitTransaction`），进程中断导致 WAL 中 3MB 未提交数据被回滚，数据库为空。

**修复建议**:
1. **事务拆分**（治本）: 将 `handleIndexRepository` 改为按文件/批次提交事务，避免单次事务过大
2. **错误传播**（治标）: 让 `handleIndexRepository` 在出错时抛出异常，确保 `setImmediate` 的 catch 能捕获并记录
3. **跳过修复**（治标）: 将图索引启动移到 `isError` 检查之前，确保向量已索引时也能触发图索引

### 7.2 整体评价

三十二次报告的 9 个 bug 全部修复，三十三次报告的代码重复也已修复。但本轮审计发现 3 个新问题，其中 2 个高优先级（图索引事务过大 + 错误静默丢弃），直接关联用户反馈的"图索引进度一直显示为 0"问题。建议优先修复这三个问题后再进行 MCP 测试。
