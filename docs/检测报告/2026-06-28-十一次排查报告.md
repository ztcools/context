## 二、新发现问题

### P2 — 代码质量

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 1 | **连锁反应** | `graph-handlers.ts:78-98` | **`mode='full'` + `specificFiles` 组合冲突**：当 `specificFiles` 非空时，`mode='full'` 仍触发 `deleteProject(project)` 删除全部项目数据，但只重新索引 `specificFiles`。其他文件的数据永久丢失。应改为：`specificFiles` 存在时仅对指定文件执行 `deleteNodesByFile`，不执行 `deleteProject`。 |
| 2 | **新发现** | `graph-store.ts:568-574` | **`buildFtsQuery` FTS5 注入风险**：用户输入中若含双引号（如搜索 `foo"bar`），`buildFtsQuery` 生成 `"foo"bar"` 导致 FTS5 查询解析失败。FTS5 中双引号需转义为 `""`。虽非 SQL 注入（参数化查询已防护），但会导致查询异常。 |

### P3 — 边界情况

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 3 | **新发现** | `openai-embedding.ts:87-116` | **`embedBatch` 无重试/限流处理**：OpenAI API 返回 429 (rate limit) 或 5xx 时直接抛出，`processChunkBuffer` 丢弃整个 chunk buffer。应添加指数退避重试（如 3 次，间隔 1s/2s/4s）。 |
| 4 | **新发现** | `context.ts:897` | **`processChunkBuffer` 期间不检查 abort signal**：`signal?.aborted` 仅在文件边界检查（line 877），但 `processChunkBuffer` 是耗时操作（embedding API 调用）。若在批量写入期间触发 abort，需等整个 batch 完成才能检测。可在 `processChunkBuffer` 内部增加 abort 检查。 |
| 5 | **新发现** | `snapshot.ts:693-724` | **`saveCodebaseSnapshot` 文件锁为 advisory**：`acquireLock()` 使用 `mkdir` 实现，是 advisory lock（非 `flock`/`fcntl`）。跨进程并发写入时，后写入者可能覆盖先写入者的数据。 |
| 6 | **新发现** | `sync.ts:160-161` | **`getIndexedCodebases()` 返回 identities 但错误日志打印了 identities**：`console.log` 打印 `indexedCodebases` 数组内容为 identities 而非 localPath，调试时可读性差。 |

## 四、结论

前十轮 73 个问题已修复（P3 除外）。第十一轮发现 6 个新问题，其中 1 个为连锁反应，5 个为新发现。

**重点关注：**
- **P2-1 连锁反应**：`deleteProject` 移入事务后，`mode='full'` + `specificFiles` 组合更危险：事务内删除全部数据 + 只重索引指定文件，commit 后其他文件数据永久丢失。修复：`specificFiles` 存在时只执行 `deleteNodesByFile`，不执行 `deleteProject`。
- **P2-2 新发现**：FTS5 查询中双引号未转义，用户搜索含引号字符串时查询失败。转义为 `""` 即可。
- **P3-3 新发现**：`embedBatch` 无重试逻辑，OpenAI 限流时丢弃整个 chunk buffer，需重新索引。

累计 79 个问题，0 个 P0。核心功能稳定，P3 问题为防御性优化，建议逐步修复。
