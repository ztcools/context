## 二、新发现问题

### P2 — 代码质量

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 1 | **新发现** | `context.ts:68-116` | **`DEFAULT_IGNORE_PATTERNS` 忽略列表不完整**：缺少 `.next/**`（Next.js）、`.nuxt/**`（Nuxt）、`vendor/**`（PHP/Go）、`.terraform/**`、`.turbo/**`（Turborepo）、`.parcel-cache/**`、`bower_components/**` 等常见构建/缓存目录。`scanFiles` 会遍历这些目录，导致索引大量无意义文件。 |
| 2 | **新发现** | `graph-handlers.ts:474-479` | **`handleDetectChanges` git diff fallback 语义不一致**（前次 P3，升级为 P2）：`git diff --name-only ${getDiffBranch}...HEAD` 失败时 fallback 到 `git diff --name-only HEAD`。前者显示「分支以来所有变更」，后者仅显示「未提交变更」。用户无感知语义差异，可能误以为所有变更文件都已检出。 |

### P3 — 边界情况

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 3 | **新发现** | `extractor.ts:280-349` | **`collectDefinitions` 无限递归无深度限制**：递归遍历 tree-sitter 语法树，无 `maxDepth` 保护。若代码含深度嵌套结构（如 1000 层嵌套函数），可能导致栈溢出。tree-sitter 理论上不会生成无限深度树，但恶意/异常代码可能触发。 |
| 4 | **新发现** | `tracer.ts:95-104` | **`bfsTrace` 中 `getNodeById` N+1 查询**：BFS 遍历中每个邻居节点都调用 `getNodeById`。对于深度为 3、平均度数 10 的调用链，约 1000 次单次查询。应批量加载。 |
| 5 | **新发现** | `graph-handlers.ts:546-553` | **`handleDeleteProject` 无事务保护**：`deleteProject` 直接调用 `this.store.deleteProject(project)`。若删除中途失败（如 SQLite 文件锁），部分节点/边可能已删除，数据不一致。应包裹在事务中。 |
| 6 | **新发现** | `sync.ts:248-250` | **`handleSyncIndex` 错误处理不完整**：`error.message.includes('Failed to query Milvus')` 时仅调用 `FileSynchronizer.deleteSnapshot` 删除 snapshot 文件，但 VDB 中的 collection 和 graph 中的 SQLite 数据未清理。三者状态不一致。 |
