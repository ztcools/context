## 二、新发现：代码重复

> **P1 — 严重代码重复**

| # | 文件 | 位置 | 说明 |
|---|------|------|------|
| 1 | `context.ts` + `synchronizer.ts` | L1349-1399 / L150-198 | **`isPatternMatch`/`matchPattern` 方法 100% 重复**（~50 行）。两个文件独立实现了相同的 glob 模式匹配逻辑，包括：目录模式匹配、根锚定模式、文件名模式等分支。 |
| 2 | `context.ts` + `synchronizer.ts` | L1388-1396 / L187-195 | **`matchesDirectoryPattern` 方法 100% 重复**（~10 行）。 |
| 3 | `context.ts` + `synchronizer.ts` | L1406-1415 / L200-211 | **`simpleGlobMatch` 方法 93% 重复**（~10 行）。唯一差异是 `synchronizer` 版本多了 `if (!text \|\| !pattern) return false` 空值守卫。 |

**影响**: 总计 ~70 行核心逻辑完全重复。修改一处需同步修改另一处，维护成本高。应抽取到 `packages/core/src/utils/` 共享模块。

**关于 `context.ts` 和 `synchronizer.ts` 重复的历史原因**: `context.ts` 用于向量索引的文件遍历（`getCodeFiles`），`synchronizer.ts` 用于 Merkle DAG 变更检测的文件遍历。两者独立实现了相同的 ignore 匹配逻辑。抽取后两处均可复用。
