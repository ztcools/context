## 四、关键问题：15 个图工具未暴露给 LLM

### 4.1 问题描述

`graph-handlers.ts` 实现了 15 个图处理器，对标 codebase-memory-mcp 的 14 个 MCP 工具：

| 处理器 | 对标 codebase-memory-mcp | 暴露状态 |
|--------|--------------------------|----------|
| `handleIndexRepository` | `index_repository` | ✅ 内部使用 (handleIndex) |
| `handleSearchGraph` | `search_graph` | ❌ 未暴露 |
| `handleTracePath` | `trace_path` | ❌ 未暴露 |
| `handleGetCodeSnippet` | `get_code_snippet` | ❌ 未暴露 |
| `handleGetGraphSchema` | `get_graph_schema` | ❌ 未暴露 |
| `handleGetArchitecture` | `get_architecture` | ❌ 未暴露 |
| `handleSearchCodeGraph` | `search_code` | ❌ 未暴露 |
| `handleDetectChanges` | `detect_changes` | ❌ 未暴露 |
| `handleListProjects` | `list_projects` | ❌ 未暴露 |
| `handleDeleteProject` | `delete_project` | ❌ 未暴露 |
| `handleIndexStatus` | `index_status` | ❌ 甚至未在 handleStatus 中使用 |
| `handleQueryGraph` | `query_graph` | ❌ 未暴露 |
| `handleManageAdr` | `manage_adr` | ❌ 未暴露 |
| `handleIngestTraces` | `ingest_traces` | ❌ 未暴露 |
| `enrichWithGraphContext` | (内部增强) | ✅ 内部使用 (handleSearchCode) |

**仅 2/15 个处理器被实际使用**，其余 13 个是**死代码**。

### 4.2 影响

LLM 无法：
- 调用 `trace_path` 追踪调用链 → 理解代码结构效率低
- 调用 `search_graph` 按标签/名称查找节点 → 只能依赖向量搜索
- 调用 `query_graph` 执行图查询 → 无法利用结构化关系
- 调用 `get_architecture` 获取架构概览 → 缺少高层视角
- 调用 `manage_adr` 管理架构决策 → 无法持久化决策
- 调用 `detect_changes` 分析变更影响 → 缺少变更影响分析

**这与项目目标"让 LLM 更加理解项目"直接矛盾**——图引擎已经构建，但 LLM 无法访问。

## 五、架构问题

### P1 — 图工具未暴露为 MCP 工具

**文件**: `packages/mcp/src/index.ts` L106-199  
**严重程度**: P1  

当前 MCP server 只注册了 4 个工具 (index/search/clear/status)，13 个已实现的图处理器未注册。LLM 无法使用图查询能力。

**修复建议**: 在 `ListToolsRequestSchema` 和 `CallToolRequestSchema` 中注册额外的图工具，并将调用路由到 `graphToolHandlers`。

### P2 — architecture.ts 缺少 Louvain 社区检测

**文件**: `packages/graph/src/architecture.ts`  
**严重程度**: P2  

codebase-memory-mcp 的 `get_architecture` 返回 Louvain 社区检测结果（功能模块聚类），当前实现仅按目录聚类，缺少基于图结构的模块发现。

### P2 — 缺少 Hybrid LSP 类型解析

**文件**: `packages/graph/src/extractor.ts`  
**严重程度**: P2  

codebase-memory-mcp 的核心优势之一是 Hybrid LSP（类型推断），当前实现仅依赖 tree-sitter 语法解析，无法处理跨文件的类型推断（如 `user.profile.display_name()` 解析到 `Profile.display_name`）。

### P2 — 缺少 MinHash 相似度检测

**严重程度**: P2  

codebase-memory-mcp 的 SIMILAR_TO 边和 MinHash+LSH 近重复检测未实现，缺少代码重复检测能力。

### P3 — 缺少跨服务链接 (gRPC/GraphQL/tRPC)

**严重程度**: P3  

当前仅支持 HTTP 调用链接，缺少 gRPC、GraphQL、tRPC 的服务检测和路由提取。

### P3 — 缺少团队共享图构件

**严重程度**: P3  

codebase-memory-mcp 的 `.codebase-memory/graph.db.zst` 团队共享机制未实现，每个开发者需独立索引。

### P3 — 缺少自动索引和文件监听

**严重程度**: P3  

codebase-memory-mcp 的 session start 自动索引和后台文件监听器未实现。

## 八、结论与建议

### 8.1 核心结论

feature/graph-engine 分支**完整保留了 main 分支原有功能**，并在其基础上增加了知识图谱引擎。但 codebase-memory-mcp 的核心精华——**让 LLM 通过图查询理解代码结构**——仅实现了 30%。

### 8.2 优先级建议

1. **P1 (最高优先级)**: 将 13 个已实现的图处理器注册为 MCP 工具，让 LLM 能够直接调用 `trace_path`、`search_graph`、`query_graph`、`get_architecture`、`manage_adr` 等
2. **P2**: 实现 Louvain 社区检测，增强架构分析能力
3. **P2**: 研究 Hybrid LSP 类型解析的可行性（TypeScript/JavaScript 优先）
4. **P3**: 添加自动索引、文件监听、团队共享等辅助功能

### 8.3 一句话总结

**图引擎已建成，但门没开。** 13 个已实现的图工具是死代码，LLM 无法使用。将图处理器暴露为 MCP 工具后，LLM 才能真正利用知识图谱理解代码结构，实现"提升开发效率、节省 token"的目标。
