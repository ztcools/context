### 1.1 核心数据流

```
用户调用 index
  ├─ handleIndex → handleIndexCodebase (向量索引)
  │   ├─ resolveCodebasePath → 路径解析
  │   ├─ getRepoIdentity → url:branch 标识
  │   ├─ hasIndex → Milvus 集合检查
  │   ├─ Context.indexCodebase → 向量索引
  │   └─ SnapshotManager → 快照更新
  │
  └─ handleIndexRepository (图索引)
      ├─ GraphExtractor.extract → AST 提取
      ├─ SqliteGraphStore → SQLite 存储
      └─ resolveCrossFileCalls → 跨文件调用解析

用户调用 search
  ├─ Context.semanticSearch → Milvus 向量搜索
  └─ enrichWithGraphContext → 图上下文增强
      ├─ 调用者/被调用者关系
      ├─ 入口点/死代码检测
      └─ 架构位置信息
```

### 1.2 隔离机制

- **身份标识**: `git remote URL + 分支名` → `url:branch`
- **向量隔离**: Milvus collection 按 `url:branch` 的 MD5 hash 命名
- **图隔离**: SQLite 表按 `project` (即 `url:branch`) 字段隔离
- **团队共享**: 同一 url+branch 的不同 clone 共享索引

## 七、结论

### 7.1 核心风险

**3 个 P0 问题必须修复：**
- **P0-1**: `ArchitectureAnalyzer.getArchitecture()` 的 edgeTypes 统计因 `getEdgesBySource(0)` placeholder 而完全失效
- **P0-2**: `handleIndex` 在图已索引时丢失图状态信息
- **P0-3**: `findNodes` 的 count 查询参数顺序不一致，可能导致统计数据错误

### 7.2 整体评价

- **架构设计**: 向量+图双引擎设计合理，url+branch 隔离方案正确，接口（Milvus + embedding）保持不变
- **代码质量**: 存在 224 行死代码、6 个已废弃方法、多处重复代码，需要清理
- **上下文精准度**: 图上下文增强只覆盖前 5 个结果，且路径匹配可能不精确，有优化空间
- **Token 效率**: 搜索结果截断长度固定 5000 字符，图上下文格式可更紧凑

### 7.3 建议优先级

1. **立即修复**: P0-1, P0-2, P0-3 (3 个功能正确性 bug)
2. **本周修复**: P1-4, P1-5, P1-6, P1-7 (4 个接口/类型问题)
3. **本迭代**: 清理死代码 (3.1 中的 4 项)，合并重复代码 (3.2 中的 3 项)
4. **下个迭代**: 性能优化 (4.1) + 上下文精准度优化 (4.2) + Token 效率优化 (4.3)
