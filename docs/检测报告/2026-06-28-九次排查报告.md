## 二、新发现问题

### P2 — 代码质量

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 1 | **连锁反应** | `handlers.ts:444-448` | **`forceReindex` 不 await 旧任务**：`oldTask.controller.abort()` 后立即删除 `indexingTasks` 条目并继续 clear+reindex。旧任务可能仍在 `processChunkBuffer` 中写入 chunk（abort 仅在文件边界检查），向已清除的 collection 写入脏数据。应像 `handleClearIndex` 那样 `await oldTask.promise` 后再继续。 |
| 2 | **新发现** | `searcher.ts:163-169` | **`enrichMatches` N+1 查询 `findNodes`**：对每个 grep 匹配调用 `this.store.findNodes()` 查找包含节点。若匹配数多（如搜索 "TODO"），会产生大量独立 SQL 查询。可改为批量查询：先收集所有 filePath，一次查询所有相关节点，再在内存中按行号匹配。 |

### P3 — 边界情况

| # | 类型 | 文件 | 问题 |
|---|------|------|------|
| 3 | **新发现** | `graph-handlers.ts:426-435` | **`handleDetectChanges` git diff 语义不一致**：`git diff --name-only ${baseBranch}...HEAD` 失败时 fallback 到 `git diff HEAD`。前者显示「baseBranch 以来所有变更」，后者仅显示「未提交变更」。若仓库有未 push 的本地提交，fallback 结果完全错误，且用户无感知。 |
| 4 | **新发现** | `graph-handlers.ts:849` | **`scanFiles` 忽略列表不完整**：缺少 `.venv`、`vendor`、`target`、`coverage`、`.cache`、`.idea`、`.vscode`、`.circleci`、`bin`、`obj`、`out`、`tmp`、`temp`、`.tox`、`.mypy_cache`、`.pytest_cache`、`.turbo` 等常见目录。全量索引时扫描大量无关文件。 |
| 5 | **新发现** | `context.ts:517-520` | **`deleteFileChunks` 中 `query` + `delete` 非原子操作**：先 query 获取 ID 列表，再 delete。若在两次调用之间 collection 被清除或 ID 被修改，delete 会静默失败（不报错但未删除任何内容）。Milvus 支持 `delete by filter` 可直接删除，避免两阶段操作。 |
| 6 | **新发现** | `graph-handlers.ts:98-104` | **`deleteNodesByFile` 仅在 `specificFiles` 或 `mode='incremental'` 时调用**：`mode='full'` 时不清除旧数据，依赖 `upsertNode` 的 `ON CONFLICT DO UPDATE` 处理。但若文件被删除或重命名，旧节点会残留。全量索引前应先清空 project 数据。 |
| 7 | **新发现** | `sync.ts:305-308` | **`setInterval` 无停止机制**：`startBackgroundSync` 创建的 `syncInterval` 无引用暴露，无法在 MCP 服务器关闭时清除。虽然进程退出时 JavaScript 会自动清理，但若需要热重启或暂停同步，无法停止定时器。 |
