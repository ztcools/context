## 二、新发现 Bug

### Bug 1: `install.sh` — `pnpm install --force` 破坏依赖一致性 🔴 高

**文件**: `install.sh:98-101`  
**严重程度**: 高 — 安装后依赖版本可能与 lockfile 不一致

**问题描述**:

```bash
# 第 98-101 行
if [ -n "$SUDO_USER" ]; then
    sudo -u "$SUDO_USER" pnpm install --force
else
    pnpm install --force
fi
```

`--force` 标志会强制 pnpm 忽略 lockfile 和缓存，重新解析所有依赖。仓库中已提交了 `pnpm-lock.yaml`（297KB），`--force` 会绕过它，导致：
- 安装的依赖版本可能与 lockfile 记录的不一致
- `better-sqlite3` 等原生模块可能被解析到不兼容的版本
- 团队成员安装后得到不同的依赖树

**修复建议**:

```bash
# 改为不带 --force 的普通安装
pnpm install
```

如果确实需要 `--force`（如清理损坏的缓存），应在注释中说明原因，并建议用户先尝试不带 `--force`。

### Bug 2: `handlers.ts` — `handleIndex` 图索引同步阻塞 MCP 响应 🔴 高

**文件**: `packages/mcp/src/handlers.ts:348-451`  
**严重程度**: 高 — 大项目索引会触发 MCP 超时 (~60s)

**问题描述**:

```typescript
// 第 353 行：向量索引是后台异步的，立即返回
const vectorResult = await this.handleIndexCodebase(args);  // ← 立即返回 "Started background indexing..."

// 第 362-448 行：图索引在主线程同步执行，阻塞 MCP 响应
if (this.graphToolHandlers) {
    if (alreadyGraphIndexed && !args.force) {
        // 增量图索引 — 同步执行
        const graphResult = await this.graphToolHandlers.handleIndexRepository({...});
    } else {
        // 全量图索引 — 同步执行，可能耗时数十秒
        const graphResult = await this.graphToolHandlers.handleIndexRepository({...});
    }
}
```

`handleIndexCodebase` 启动后台向量索引后立即返回，但图索引（`handleIndexRepository`）在同一个 `handleIndex` 调用中**同步执行**。对于大型仓库，图索引的 `scanFiles` + `extract` + `resolveCrossFileCalls` 可能需要 30-120 秒，远超 MCP 协议的默认超时（~60s）。

此外，`handleIndexRepository` 内部使用 `fs.readFileSync` 同步读取每个文件（第 132 行），在 Node.js 事件循环中大量同步 I/O 会阻塞所有并发请求。

**修复建议**:
- 将图索引也改为后台执行，与向量索引一致
- 或者在 `handleIndex` 中先返回 "started" 响应，再异步执行图索引

### Bug 3: `handlers.ts` — 文本解析 `handleDetectChanges` 输出，极其脆弱 🔴 高

**文件**: `packages/mcp/src/handlers.ts:372-382`  
**严重程度**: 高 — 增量索引可能静默失败

**问题描述**:

```typescript
// 第 372-382 行
const changesResult = this.graphToolHandlers.handleDetectChanges({
    project,
});
const changesText = changesResult.content[0]?.text || '';
const changedMatch = changesText.match(/Changed files:\s*\n([\s\S]*?)(?:\n\n|$)/);
if (changedMatch) {
    changedFiles = changedMatch[1]
        .split('\n')
        .map(l => l.trim().replace(/^-\s*/, ''))
        .filter(l => l.length > 0);
}
```

用正则表达式从 `handleDetectChanges` 的**格式化文本输出**中提取文件路径列表。这是格式耦合——`handleDetectChanges` 的文本格式（`graph-handlers.ts:516-519`）:

```typescript
lines.push(`Changed files: ${changedFiles.length}`);
for (const file of changedFiles) {
    lines.push(`  ${file}`);
}
```

如果 `handleDetectChanges` 的格式有任何变化（如改缩进、加前缀、改标点），正则匹配就会失败，`changedFiles` 变为空数组，`handleIndex` 的增量索引逻辑会认为"没有变更"从而跳过图索引，且**不会报告任何错误**。

**修复建议**:
- 不要解析文本输出。在 `GraphToolHandlers` 中暴露一个返回结构化数据的方法（如 `detectChangedFiles(): string[]`），由 `handleIndex` 直接调用
- 或在 `handleDetectChanges` 的返回值中附加结构化字段

### Bug 4: `snapshot.ts` — `saveCodebaseSnapshot` 读-合并-写存在 TOCTOU 竞态 🟡 中

**文件**: `packages/mcp/src/snapshot.ts:718-730`  
**严重程度**: 中 — 多进程同时保存时可能丢失数据

**问题描述**:

```typescript
// 第 719-730 行：读取磁盘 → 合并 → 写回
try {
    if (fs.existsSync(this.snapshotFilePath)) {
        const diskData = fs.readFileSync(this.snapshotFilePath, 'utf8');
        const diskSnapshot = JSON.parse(diskData);
        if (this.isV2Format(diskSnapshot)) {
            for (const [diskKey, diskInfo] of Object.entries(diskSnapshot.codebases)) {
                this.mergeExternalEntry(diskKey, diskInfo as CodebaseInfo);
            }
        }
    }
} catch (mergeError) { ... }
```

虽然有 `acquireLock`（基于目录 mkdir 的互斥锁），但快照文件没有使用原子写入（先写临时文件再 `rename`），写入过程中崩溃会留下损坏的 JSON 文件。两个独立 MCP 进程同时保存时，先写后读的进程可能覆盖另一个进程的写入。

**修复建议**:
- 使用 `fs.writeFileSync(tmpPath, ...)` + `fs.renameSync(tmpPath, snapshotPath)` 实现原子写入
- 读取时使用 try-catch 包裹 `JSON.parse` 以处理损坏的文件

### Bug 5: `graph-handlers.ts` — `findRepoPath` 搜索范围太窄 🟢 低

**文件**: `packages/mcp/src/graph-handlers.ts:889-894`  
**严重程度**: 低 — 变更检测在非标准路径下失败

**问题描述**:

```typescript
// 第 889-894 行
const searchPaths = [
    process.cwd(),
    path.join(homeDir, 'deploy'),
    homeDir,
];
```

`findRepoPath` 只在 `cwd`、`~/deploy`、`~` 三个位置搜索仓库。用户 clone 到其他路径（如 `~/projects/xxx`、`~/work/xxx`）时返回 `null`，导致 `handleDetectChanges` 中无法执行 `git diff`，增量索引判断失败。

**修复建议**:
- 从 `codebaseInfoMap` 的 `localPath` 字段直接获取仓库路径，而不是在文件系统中搜索
- 或者扩展搜索路径，至少包含 `~/projects`、`~/code`、`~/src` 等常见目录

### Bug 6: `sync.ts` — `isSyncing` 和锁获取之间无原子性保证 🟢 低

**文件**: `packages/mcp/src/sync.ts:173-184`  
**严重程度**: 低 — 逻辑冗余，状态不一致

**问题描述**:

```typescript
// 第 173-184 行
if (this.isSyncing) {                              // ① 检查
    return;
}
if (!this.acquireGlobalSyncLock()) {                // ② 获取锁
    return;
}
this.isSyncing = true;                              // ③ 设置标志
```

在单进程内，①和②之间没有原子性保证。如果两个并发的 `handleSyncIndex` 调用同时通过 `isSyncing` 检查，它们都会尝试获取全局锁。虽然全局锁最终只让一个获胜，但 `isSyncing` 标志实际上是无用的冗余检查。

**修复建议**:
- 将 `isSyncing` 的检查/设置合并到 `acquireGlobalSyncLock` 中，或直接移除 `isSyncing`，完全依赖全局锁

### Bug 7: `graph-handlers.ts` — 图扫描的 `IGNORE_DIRS` 与向量索引 ignore patterns 不一致 🟢 低

**文件**: `packages/mcp/src/graph-handlers.ts:35-40` vs `packages/core/src/context.ts:69-129`  
**严重程度**: 低 — 向量和图索引覆盖的文件集合可能不同

**问题描述**:

图索引使用硬编码的目录集合：

```typescript
// graph-handlers.ts:35-40
const IGNORE_DIRS = new Set([
    'node_modules', '.git', 'dist', 'build', '.next', '__pycache__',
    '.venv', 'vendor', 'target', 'coverage', '.cache', '.idea', '.vscode',
    '.circleci', 'bin', 'obj', 'out', 'tmp', 'temp', '.tox',
    '.mypy_cache', '.pytest_cache', '.turbo', '.angular', '.nuxt',
]);
```

向量索引使用 `Context` 的 ignore patterns（`DEFAULT_IGNORE_PATTERNS` + `.gitignore` + `.contextignore`）。不一致场景：
- 用户在 `.gitignore` 中排除了 `generated/`，向量索引会跳过，但图索引会扫描
- 图索引忽略 `.circleci`，但向量索引不会（除非在 `.gitignore` 中）
- 图索引忽略 `.venv`，但向量索引的默认 patterns 中没有 `.venv`

**修复建议**:
- 让图索引的 `scanFiles` 复用 `Context.getEffectiveIgnorePatterns()` 的结果
- 或在 `GraphToolHandlers` 构造时接收 ignore patterns 参数

### Bug 8: `graph-handlers.ts` — `resolveCrossFileCalls` 一次性加载所有节点 🟢 低

**文件**: `packages/mcp/src/graph-handlers.ts:797`  
**严重程度**: 低 — 大型项目性能问题

**问题描述**:

```typescript
const allNodes = this.store.findNodes({ project, limit: MAX_CROSS_FILE_NODES }); // 200,000
```

对于大型 monorepo，一次性加载 20 万个节点到内存构建全局注册表，然后加载所有 IMPORTS 边，内存压力和时间复杂度较高。

**修复建议**:
- 分批加载节点，按文件粒度处理
- 或只在需要时查询特定 importedName 的候选节点

### Bug 9: `install.sh` — `claude mcp add` 错误被 `2>/dev/null` 完全吞掉 🟢 低

**文件**: `install.sh:143-152`  
**严重程度**: 低 — 用户体验问题

**问题描述**:

```bash
claude mcp add claude-context -s user ... -- node "$MCP_ENTRY" 2>/dev/null
```

所有错误输出被 `2>/dev/null` 丢弃。如果 `claude mcp add` 失败（如 Claude Code 未安装、权限不足），用户看不到错误信息，只能看到 fallback 消息但不知道根本原因。

**修复建议**:
- 移除 `2>/dev/null`，让错误信息可见
- 或者将 stderr 重定向到日志文件而不是丢弃
