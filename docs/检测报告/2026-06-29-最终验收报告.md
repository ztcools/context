## 一、项目概述

本项目基于开源 claude-context（main 分支），汲取 codebase-memory-mcp 的知识图谱精华，在保留原有向量索引能力的基础上，增加了 SQLite 知识图谱引擎，使 LLM 通过 MCP 工具调用时能获得更丰富的代码结构上下文。

## 二、在 main 分支基础上做的改进

### 2.1 新增：知识图谱引擎 (`packages/graph/`)

| 模块 | 功能 | 对标 codebase-memory-mcp |
|------|------|--------------------------|
| `graph-store.ts` | SQLite 图存储（FTS5 全文索引、节点/边 CRUD、批量查询） | Knowledge Graph Store |
| `extractor.ts` | tree-sitter AST 解析（TS/Python/Java/Go 等） | Tree-sitter Parsing |
| `tracer.ts` | BFS 调用链追踪 | trace_path |
| `searcher.ts` | 结构化图搜索 + BM25 全文搜索 | search_graph |
| `architecture.ts` | 架构分析（入口点检测、目录聚类、内聚度计算） | get_architecture |
| `types.ts` | 图节点/边类型定义（Function/Class/Route/Resource + CALLS/IMPORTS/HTTP_CALLS 等） | Graph Schema |

### 2.2 新增：MCP 图处理器 (`packages/mcp/src/graph-handlers.ts`)

15 个图处理器，完整对标 codebase-memory-mcp 的 14 个 MCP 工具：
- `handleIndexRepository`, `handleSearchGraph`, `handleTracePath`, `handleGetCodeSnippet`, `handleGetGraphSchema`, `handleGetArchitecture`, `handleSearchCodeGraph`, `handleDetectChanges`, `handleListProjects`, `handleDeleteProject`, `handleIndexStatus`, `handleQueryGraph`, `handleManageAdr`, `handleIngestTraces`

### 2.3 改造：4 个统一 MCP 工具深度接入图引擎

**核心设计理念**：保持 4 个 MCP 工具接口不变，内部将图处理器深度接入，让 LLM 无需学习新工具即可获得增强能力。

| 工具 | 改造内容 | 效果 |
|------|----------|------|
| `index` | 集成 `detect_changes` 实现智能增量检测 | 首次全量索引，后续自动检测变更，仅索引变更文件 |
| `search` | 三层图增强：调用关系 + 调用链追踪 + 架构摘要；向量无结果时 fallback 到图搜索 | LLM 一次调用获得代码片段 + 谁调它 + 它调谁 + 完整调用链 + 所属模块 |
| `clear` | 不变 | 向量 + 图联合清除 |
| `status` | 集成 `get_architecture` 显示完整架构图 | LLM 可获得入口点、模块聚类、内聚度等架构信息 |

### 2.4 保留：main 分支所有原有功能

## 四、发现 1 个观察项（非阻塞）

### P3 — `enrichWithGraphContextDeep` 的 `handleTracePath` 调用可更精确

**位置**: `handlers.ts` L1554  
**现象**: 传递 `function_name: r.node.name`（如 `authenticate`）而非 `r.node.qualifiedName`（如 `src/auth.ts::authenticate`）。  
**影响**: 极少情况下，若不同文件有同名函数，`handleTracePath` 内部 `namePattern` LIKE 搜索可能匹配到错误的函数。  
**触发条件**: 同一项目中存在跨文件的同名函数，且两者都在搜索结果中。  
**修复**: 改为 `function_name: r.node.qualifiedName`  
**上线建议**: 不影响上线。概率极低，可择机修复。
