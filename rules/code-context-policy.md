# 代码上下文策略 · claude-context (search / index / status)

**适用守卫**：本节仅在当前会话可用 `search` / `index` / `status` 这组 MCP 工具（claude-context）时生效。若工具列表中没有它们，请**完全忽略本节**——它是空操作。

## 这个工具独有的价值（决定何时该用它）
`search` 不是"另一个 grep"。它做两件 grep/read 做不到的事：
1. **语义检索**：按*意图*找代码（稠密向量 + BM25 混合），你不知道确切函数名/关键词时也能命中。
2. **调用图上下文**：每个命中附带 调用者`←` / 被调用者`→` / `[unused]`(死代码) / `[entry]`(入口)，以及跨文件、跨服务的调用链和架构摘要（`CALLS/IMPORTS/EXTENDS/HTTP_CALLS/CROSS_*`、路由节点）。

**判定用不用 search，看两个问题，而不是看项目大小：**
- **我已经知道确切的字符串/符号/文件位置吗？** 知道 → 直接 grep/read（search 只会更慢）。不知道 → search。
- **我问的是"关系/结构"吗？**（谁调用它、调用链、改动影响面、死代码、入口、路由、跨服务）→ **只有 search 能答**，优先用它。

## search 明显更优的场景（**优先用**，这里它远胜不装 MCP）
- **理解陌生代码/流程**："X 功能怎么实现的"、"请求怎么从入口走到 DB"——一次 search 拿到片段 + 调用链，替代十几次盲读。
- **重构前评估影响面**：先 search 目标符号，看 `←调用者` 列表和调用链，判断改了会波及谁。**动手改之前必做。**
- **Bug 根因定位**：search 症状/报错语义 → 顺 `→被调用者`/调用链跨文件追到根因。
- **死代码/入口排查**：`[unused]` / `[entry]` 标记直接给出，grep 无法判断。
- **找现有模式/约定**：加功能前 search 类似实现，照着写，保持一致。

## 何时**不要**用 search（用了反而不如不装 MCP）
- **已知精确 token/符号/路径**：用 Grep/Read，精确、即时、零成本。
- **非 AST 支持语言**（图不生效，只剩弱语义）：Ruby、PHP、Kotlin、Swift、Vue/模板、以及配置/YAML/JSON/锁文件/Markdown 等——优先 Grep/Read。
  - AST/图**支持**的语言：JS/TS、Python、Java、C/C++、Go、Rust、C#、Scala。
- **要逐字完整内容**（编辑整文件、格式敏感）：直接 Read。
- **枚举文件、按名/扩展名找**：用 glob/Grep。

## 用法（拿到高效上下文 + 省 token 的关键）
1. `search "<用自然语言描述意图>"` 定位 → 得到排序 `file:line` + 调用图。
2. 只对真正要改/要读的地方，用 **Read 的 offset/limit 定点读那几行区间**，不要整文件通读——这是省 token 的核心动作。
3. 首轮不准就换更聚焦的 query 再搜；一个概念一条 query，多部分任务拆多次。
4. 若 search 提示未索引：对支持的语言 `index` 一次（一次性成本，之后增量自动）；对极小项目或非支持语言，直接 read/grep 别索引。
5. search 没有有用结果就**立刻回退** grep/read，不要死循环。

## 规模只作次要修正（不作主判据）
- 同样"不完全确定关键词"时：仓库越大越倾向 search（grep 在大库里噪声太多）；单目录小模块直接 read/grep 即可。
- 项目大小从不单独决定用不用 search——**问题性质和语言支持才是主判据**。
